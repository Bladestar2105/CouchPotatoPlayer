So the problem is exactly this:
For Pods that use `s.ios.dependency 'Flutter'` (like `path_provider_foundation`, `media_kit_video`), CocoaPods generates `.xcconfig` files with:
`FRAMEWORK_SEARCH_PATHS[sdk=iphoneos*] = .../out/ios_release`
But we are building for `appletvos`!
So `FRAMEWORK_SEARCH_PATHS` for `appletvos` DOES NOT get `.../out/ios_release`!
And `tvos/Podfile` tries to "fix" this by manually adding `-F .../ios/Flutter`!
But `Flutter.framework` is NOT in `ios/Flutter` during GitHub Actions CI build! It is in `out/ios_release`!
Because we set `FLUTTER_LOCAL_ENGINE=${{ github.workspace }}/out/ios_release`!

HOW TO FIX THIS?!
In `tvos/Podfile`, we need to read `FLUTTER_LOCAL_ENGINE` or extract the search path from the `.xcconfig` generated by CocoaPods!
OR, simpler:
We can manually copy the inherited `iphoneos` search paths to `appletvos` search paths in the `.xcconfig`!
No, `tvos/Podfile` can do this:
```ruby
      flutter_dir = ENV['FLUTTER_LOCAL_ENGINE'] || (File.expand_path('..', $PROJECT_DIR) + '/ios/Flutter')
```
Wait! If `ENV['FLUTTER_LOCAL_ENGINE']` is set, we can use it!
BUT wait! `ENV['FLUTTER_LOCAL_ENGINE']` is NOT the path to the framework, it's the path to the engine directory (which contains `Flutter.xcframework` or `Flutter.framework`).
If `ENV['FLUTTER_LOCAL_ENGINE']` is used, the `-F` path should be `ENV['FLUTTER_LOCAL_ENGINE']`!

Let's check what `FLUTTER_LOCAL_ENGINE` is in `.github/workflows/build-apps.yml`:
```yaml
          export FLUTTER_LOCAL_ENGINE=${{ github.workspace }}/out/ios_release
```
And `out/ios_release` contains the built custom engine!
If we modify `tvos/Podfile` to check `ENV['FLUTTER_LOCAL_ENGINE']`, it will append the correct directory to `FRAMEWORK_SEARCH_PATHS`!
