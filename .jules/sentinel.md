## 2024-05-15 - Information Exposure via Fetch Error Messages
**Vulnerability:** Network fetch operations threw original error messages (`e.message` or `fetchError.message`) to the frontend when Xtream API authentication or M3U list downloads failed. Because Xtream API passes credentials as query parameters in the URL (`?username=...&password=...`), these plaintext credentials could be leaked on the UI inside the error message string from the failed `fetch` request.
**Learning:** Raw network error messages often echo the requested URL. When APIs embed sensitive data within the URL, propagating the raw error message to the UI creates an Information Exposure vulnerability.
**Prevention:** Never throw or pass raw network error messages directly to the UI layer when connecting to APIs that use credentials in their URLs. Always log the original error using `console.error` for debugging observability, and throw a sanitized, generic error message (e.g., "Network error occurred") to the user.
## 2024-05-15 - SSRF & Malicious Protocol Injection via Unvalidated URLs
**Vulnerability:** The application allowed users to input unvalidated server URLs (for both M3U and Xtream profiles) without enforcing a specific URL scheme (like `http://` or `https://`). This could allow attackers to input arbitrary protocols (e.g., `file://`, `intent://`, `javascript:`) potentially leading to Server-Side Request Forgery (SSRF) when the application fetches from these endpoints, or enabling local file read if the underlying fetch library mishandles local schemes.
**Learning:** All user-supplied URLs that the application uses to make network requests must be strictly validated. Relying purely on network errors or implicit fetch behavior is unsafe and can lead to unintended protocol execution.
**Prevention:** Always validate user-provided URLs by verifying they start with safe, expected schemes (e.g., strictly `http://` or `https://`) before passing them to any network fetch functions or saving them to application state.

## 2026-03-23 - Insecure Random Number Generation for Identifiers
**Vulnerability:** Using `Math.random().toString()` to generate unique identifiers (such as EPG program IDs) is insecure because `Math.random()` is not cryptographically secure and is prone to collisions, especially with large datasets.
**Learning:** Non-cryptographic PRNGs like `Math.random()` can produce predictable sequences and are unsuitable for generating unique IDs in applications processing high volumes of data. Modern environments provide the `crypto.randomUUID()` API for this purpose.
**Prevention:** Use `crypto.randomUUID()` when available to generate secure, unique identifiers. Implement a defensive check to fallback gracefully in older or non-secure environments: `(typeof crypto !== 'undefined' && crypto.randomUUID) ? crypto.randomUUID() : Math.random().toString()`.

## 2024-05-15 - Information Exposure in Proxy Server Logs
**Vulnerability:** The proxy server logs the full error message (`e.message`) when a network fetch operation fails. Because the URL passed to the proxy server may contain sensitive credentials (e.g. `username` and `password` for Xtream API) as query parameters, and raw network error messages often include the original requested URL, these plain-text credentials could be leaked directly into the server's stdout logs.
**Learning:** Raw network error messages often echo the requested URL. Propagating the raw error message to the server logs without sanitization creates an Information Exposure vulnerability when APIs embed sensitive data within the URL.
**Prevention:** Never log raw network error messages directly to the server logs when proxying requests to APIs that use credentials in their URLs. Always apply sanitization (e.g., using a regex replacement to mask sensitive query parameters) to the error message before logging it using `console.error`.

## 2024-06-12 - Information Exposure via URL Authority in Proxy Logs
**Vulnerability:** The proxy server masked `username` and `password` query parameters in logs, but failed to mask Basic Auth credentials contained in the URL authority (e.g., `http://user:pass@host`). Calling `u.toString()` on a `URL` object or using an incomplete regex for fallback logging leaked these credentials in plain text to the server logs.
**Learning:** Credentials in URLs can appear in both query parameters and the authority section. Masking one does not automatically protect the other. The `URL` API in Node.js includes authority credentials in its string representation unless they are explicitly cleared.
**Prevention:** When logging URLs that may contain credentials, explicitly mask or clear both the `username` and `password` properties of the `URL` object. For regex-based sanitization, ensure the pattern accounts for the `://user:pass@` format in addition to query parameters.
