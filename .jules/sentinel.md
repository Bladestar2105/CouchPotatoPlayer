## 2024-05-15 - Information Exposure via Fetch Error Messages
**Vulnerability:** Network fetch operations threw original error messages (`e.message` or `fetchError.message`) to the frontend when Xtream API authentication or M3U list downloads failed. Because Xtream API passes credentials as query parameters in the URL (`?username=...&password=...`), these plaintext credentials could be leaked on the UI inside the error message string from the failed `fetch` request.
**Learning:** Raw network error messages often echo the requested URL. When APIs embed sensitive data within the URL, propagating the raw error message to the UI creates an Information Exposure vulnerability.
**Prevention:** Never throw or pass raw network error messages directly to the UI layer when connecting to APIs that use credentials in their URLs. Always log the original error using `console.error` for debugging observability, and throw a sanitized, generic error message (e.g., "Network error occurred") to the user.
## 2024-05-15 - SSRF & Malicious Protocol Injection via Unvalidated URLs
**Vulnerability:** The application allowed users to input unvalidated server URLs (for both M3U and Xtream profiles) without enforcing a specific URL scheme (like `http://` or `https://`). This could allow attackers to input arbitrary protocols (e.g., `file://`, `intent://`, `javascript:`) potentially leading to Server-Side Request Forgery (SSRF) when the application fetches from these endpoints, or enabling local file read if the underlying fetch library mishandles local schemes.
**Learning:** All user-supplied URLs that the application uses to make network requests must be strictly validated. Relying purely on network errors or implicit fetch behavior is unsafe and can lead to unintended protocol execution.
**Prevention:** Always validate user-provided URLs by verifying they start with safe, expected schemes (e.g., strictly `http://` or `https://`) before passing them to any network fetch functions or saving them to application state.
## 2024-05-18 - Information Exposure via URL Logging
**Vulnerability:** The application was logging raw URLs (in `console.log`) that contained sensitive query parameters (`username` and `password`) used for authenticating with the Xtream API during EPG and proxy requests. This leads to Information Exposure in the application or proxy server logs.
**Learning:** URL logging often inherently captures any sensitive parameters passed as part of a query string.
**Prevention:** Always parse and sanitize URLs before logging to strip or mask out sensitive query string parameters, such as API keys, usernames, or passwords.
