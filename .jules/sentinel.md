## 2024-05-15 - Information Exposure via Fetch Error Messages
**Vulnerability:** Network fetch operations threw original error messages (`e.message` or `fetchError.message`) to the frontend when Xtream API authentication or M3U list downloads failed. Because Xtream API passes credentials as query parameters in the URL (`?username=...&password=...`), these plaintext credentials could be leaked on the UI inside the error message string from the failed `fetch` request.
**Learning:** Raw network error messages often echo the requested URL. When APIs embed sensitive data within the URL, propagating the raw error message to the UI creates an Information Exposure vulnerability.
**Prevention:** Never throw or pass raw network error messages directly to the UI layer when connecting to APIs that use credentials in their URLs. Always log the original error using `console.error` for debugging observability, and throw a sanitized, generic error message (e.g., "Network error occurred") to the user.
## 2024-05-15 - SSRF & Malicious Protocol Injection via Unvalidated URLs
**Vulnerability:** The application allowed users to input unvalidated server URLs (for both M3U and Xtream profiles) without enforcing a specific URL scheme (like `http://` or `https://`). This could allow attackers to input arbitrary protocols (e.g., `file://`, `intent://`, `javascript:`) potentially leading to Server-Side Request Forgery (SSRF) when the application fetches from these endpoints, or enabling local file read if the underlying fetch library mishandles local schemes.
**Learning:** All user-supplied URLs that the application uses to make network requests must be strictly validated. Relying purely on network errors or implicit fetch behavior is unsafe and can lead to unintended protocol execution.
**Prevention:** Always validate user-provided URLs by verifying they start with safe, expected schemes (e.g., strictly `http://` or `https://`) before passing them to any network fetch functions or saving them to application state.

## 2026-03-23 - Insecure Random Number Generation for Identifiers
**Vulnerability:** Using `Math.random().toString()` to generate unique identifiers (such as EPG program IDs) is insecure because `Math.random()` is not cryptographically secure and is prone to collisions, especially with large datasets.
**Learning:** Non-cryptographic PRNGs like `Math.random()` can produce predictable sequences and are unsuitable for generating unique IDs in applications processing high volumes of data. Modern environments provide the `crypto.randomUUID()` API for this purpose.
**Prevention:** Use `crypto.randomUUID()` when available to generate secure, unique identifiers. Implement a defensive check to fallback gracefully in older or non-secure environments: `(typeof crypto !== 'undefined' && crypto.randomUUID) ? crypto.randomUUID() : Math.random().toString()`.

## 2024-05-15 - Information Exposure in Proxy Server Logs
**Vulnerability:** The proxy server logs the full error message (`e.message`) when a network fetch operation fails. Because the URL passed to the proxy server may contain sensitive credentials (e.g. `username` and `password` for Xtream API) as query parameters, and raw network error messages often include the original requested URL, these plain-text credentials could be leaked directly into the server's stdout logs.
**Learning:** Raw network error messages often echo the requested URL. Propagating the raw error message to the server logs without sanitization creates an Information Exposure vulnerability when APIs embed sensitive data within the URL.
**Prevention:** Never log raw network error messages directly to the server logs when proxying requests to APIs that use credentials in their URLs. Always apply sanitization (e.g., using a regex replacement to mask sensitive query parameters) to the error message before logging it using `console.error`.

## 2024-06-12 - Information Exposure via URL Authority in Proxy Logs
**Vulnerability:** The proxy server masked `username` and `password` query parameters in logs, but failed to mask Basic Auth credentials contained in the URL authority (e.g., `http://user:pass@host`). Calling `u.toString()` on a `URL` object or using an incomplete regex for fallback logging leaked these credentials in plain text to the server logs.
**Learning:** Credentials in URLs can appear in both query parameters and the authority section. Masking one does not automatically protect the other. The `URL` API in Node.js includes authority credentials in its string representation unless they are explicitly cleared.
**Prevention:** When logging URLs that may contain credentials, explicitly mask or clear both the `username` and `password` properties of the `URL` object. For regex-based sanitization, ensure the pattern accounts for the `://user:pass@` format in addition to query parameters.

## 2026-10-27 - Information Exposure in Python Proxy Server Logs
**Vulnerability:** The Python proxy server (`proxy-server.py`) used the default `log_message` implementation of `BaseHTTPRequestHandler`, which logs the full raw request line to `sys.stderr`. Because IPTV requests proxied through it often contain sensitive credentials either in the URL authority (`http://user:pass@host`) or as query parameters (`username=...&password=...`), these credentials were leaked in plaintext to the server logs.
**Learning:** The default logging mechanisms in standard library HTTP handlers (like Python's `http.server`) are often overly verbose and do not sanitize sensitive data from URLs. When a proxy server handles URLs containing credentials, relying on default request logging creates an Information Exposure vulnerability.
**Prevention:** Always override default logging methods (e.g., `log_message` in Python's `BaseHTTPRequestHandler`) when handling sensitive URLs. Apply explicit sanitization using regular expressions to mask both query parameters (`username`, `password`) and URL authority credentials (`://user:pass@`) before writing the request line to any log output.

## 2026-04-01 - Log Injection / CRLF Vulnerability in Node Proxy Logs
**Vulnerability:** The Node.js proxy server (`proxy-server.js`) was vulnerable to Log Injection attacks. Unsanitized strings containing user input—specifically the requested URL (`targetUrl`) and error messages (`e.message`) resulting from failed URL parsing or fetch operations—were passed directly to `console.log` and `console.error`. If a malicious URL contained carriage return (`\r`) or line feed (`\n`) characters, these were rendered directly into the server logs, allowing attackers to inject forged log entries.
**Learning:** Raw input strings, even when partially masked for credentials, can still contain control characters that manipulate the log file format. Trusting error messages from standard APIs (like `URL` parsing) is also unsafe if the error message reflects the raw, unsanitized input.
**Prevention:** Implement a global sanitization helper (e.g., `msg.replace(/\r/g, '').replace(/\n/g, '')`) and apply it to *all* variables passed to logging functions (`console.log`, `console.error`) that incorporate user input, whether directly or via generated error strings.
## 2025-05-22 - Insecure Random Number Generation for IDs
**Vulnerability:** The application used `Date.now().toString()` to generate unique identifiers for newly created IPTV profiles. Since `Date.now()` provides the current timestamp in milliseconds, creating profiles rapidly (e.g., in automated scripts or bulk operations) could result in colliding IDs, leading to state corruption and unpredictable behavior.
**Learning:** Time-based generation without cryptographic uniqueness guarantees predictable sequences and collision risks. Modern environments provide `crypto.randomUUID()` for generating secure UUIDv4 identifiers.
**Prevention:** Always use `crypto.randomUUID()` to generate unique identifiers. For environments with varying compatibility (like React Native), implement the defensive fallback: `(typeof crypto !== 'undefined' && crypto.randomUUID) ? crypto.randomUUID() : Date.now().toString()`.

## 2024-05-23 - Hardcoded TMDB API Key in MediaInfoScreen
**Vulnerability:** The application contained a hardcoded TMDB API key placeholder (`'YOUR_API_KEY_HERE'`) in the `MediaInfoScreen.tsx` component. Hardcoding secrets in source code leads to information exposure and prevents secure management/rotation of credentials.
**Learning:** Hardcoding API keys or secrets in the UI layer (or any source file) is a security risk. It makes the secret accessible to anyone with access to the source code or the bundled application.
**Prevention:** Never hardcode secrets in source code. Use environment variables (prefixed with `EXPO_PUBLIC_` for Expo) for build-time defaults and provide a UI for users to supply their own keys, persisting them securely in local storage (like `AsyncStorage`). Retrieve secrets via a centralized configuration or context provider.

## 2024-05-24 - Server-Side Request Forgery (SSRF) in Python Proxy Server
**Vulnerability:** The Python proxy server (`proxy-server.py`) forwarded requests to any URL provided by the user without validating the destination IP address. This allowed an attacker to supply URLs pointing to internal or private IP addresses (e.g., `127.0.0.1`, `169.254.169.254`), resulting in Server-Side Request Forgery (SSRF) and potential exposure of internal services or cloud metadata.
**Learning:** Checking URL schemes (`http://` vs `file://`) is insufficient to prevent SSRF. Attackers can use valid schemes but direct the request to restricted internal IP addresses, bypassing network boundaries.
**Prevention:** Always resolve the hostname of the target URL to its IP address (using `socket.getaddrinfo`) and validate the resulting IP against a strict denylist of private, loopback, and link-local ranges (e.g., using `ipaddress`) before initiating the proxy request.
## 2025-10-18 - Information Exposure via Unsanitized Exception Details in API Responses
**Vulnerability:** The Python proxy server returned raw exception messages (`str(e)`) to the client when DNS resolution or network requests failed. This could potentially expose internal hostnames, IP details, or DNS configurations. Additionally, manually formatting `str(e)` within an f-string to construct a JSON response was prone to generating malformed JSON if the exception message contained unescaped quotes or newlines.
**Learning:** Exposing internal error states directly to clients breaches the principle of failing securely and can leak infrastructure details. Furthermore, string interpolation is an unsafe method for constructing structured formats like JSON, as it is vulnerable to injection.
**Prevention:** Fail securely by returning generic error messages in API responses (e.g., "DNS resolution failed"). Always use established serialization libraries like `json.dumps()` to format JSON responses instead of string interpolation.
